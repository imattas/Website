## Recon / Initial Analysis

Before digging into the crypto, I tried the obvious: submitting C2 URLs I already knew from earlier tasks.

```
https://203.0.113.42/module          ŌåÉ Task 4's C2 IP
https://203.0.113.108:26535/module   ŌåÉ Task 2's DNS-poisoned IP
```

Both wrong. This task wasn't about reusing existing intelligence ŌĆō it required breaking the malware's encryption to uncover a *new* C2 endpoint.

### Reversing the Comms Class

The `extracted.so` is the shared library that Task 4's malware downloads and `dlopen`s. It links only against `libcrypto.so.3` (not `libssl`) and exports 13 methods of a C++ `Comms` class: `gen_key`, `connect_to_server`, `recv_rsa_pubkey`, `send_aes_keys`, `rsa_encrypt`, `send_message`, `is_correct_response`, `application_handshake`, and `full_handshake`. Loading it in IDA revealed how the class handles all encrypted communication with the C2 server.

#### Constructor: Double AES-128-ECB

The `Comms::Comms` constructor sets up the encryption:

```
Comms::Comms():
    encA = EVP_EncryptInit_ex(..., EVP_aes_128_ecb, keyA)
    encB = EVP_EncryptInit_ex(..., EVP_aes_128_ecb, keyB)
    decA = EVP_DecryptInit_ex(..., EVP_aes_128_ecb, keyA)
    decB = EVP_DecryptInit_ex(..., EVP_aes_128_ecb, keyB)
```

Four OpenSSL contexts, two keys. Encryption chains as **EncA then EncB**; decryption reverses as **DecB then DecA**. This is textbook double encryption ŌĆō and it has a textbook weakness.

---

## Vulnerability / Observation

Each key is generated by `gen_key ŌåÆ generate_key`, and this is where the malware author made a critical mistake:

1. Read 32 bytes from `/dev/random` (good so far)
2. Compute `H = HMAC(SHA3-256, key="ABCD1233321", data=random32)` (still okay)
3. XOR the upper 16 bytes of the random data with the first 16 bytes of H (fine)
4. Extract the high 64 bits of the result
5. **Right-shift by 38 bits** (0x26)
6. Write the shifted value back as the key, zero-pad the rest

Step 5 is the catastrophe. A right-shift by 38 bits on a 64-bit value leaves only **26 bits of entropy**. The resulting key looks like:

```
keyA = [4 bytes of data] 00 00 00 00 00 00 00 00 00 00 00 00
```

Each AES-128 key has an effective keyspace of 2^26 ŌĆō about 67 million possibilities. That's brute-forceable on a laptop. Combined with double encryption (normally 2^256), this collapses the whole scheme to a meet-in-the-middle attack of ~2^27 work.

---

## Exploitation / Solution

### Meet-in-the-Middle Attack

Double encryption with two independent keys is vulnerable to a classic [meet-in-the-middle attack](https://en.wikipedia.org/wiki/Meet-in-the-middle_attack). Instead of searching 2^26 ├Ś 2^26 = 2^52 combinations, MITM reduces it to 2^26 + 2^26 = 2^27 operations plus a lookup table.

#### Known Plaintext

The protocol uses a magic/version header followed by command strings. From the binary's string checks, the first 16-byte block of a connection acknowledgment is:

```
P = de c0 de c0 ff ee 52 45 51 43 4f 4e 4e 5f 4f 4b
                       R  E  Q  C  O  N  N  _  O  K
```

The corresponding ciphertext block (captured from the malware's communications):

```
C = bb 39 35 59 c5 82 af a5 0b 2e 57 c2 cf 5b 2d c7
```

#### The Attack

Since encryption is `C = EncB(EncA(P))`, there exists an intermediate value `M = EncA(P) = DecB(C)`. The attack finds both keys by meeting at M:

**Phase 1 ŌĆō Build the forward table:**

```
For each stateA in [0, 2^26):
    keyA = LE64(stateA) || 8├Ś0x00
    X = AES_ECB_Encrypt(keyA, P)
    Store tag(X) ŌåÆ stateA in hash table
```

**Phase 2 ŌĆō Probe from the ciphertext side:**

```
For each stateB in [0, 2^26):
    keyB = LE64(stateB) || 8├Ś0x00
    Y = AES_ECB_Decrypt(keyB, C)
    If tag(Y) found in table:
        Verify full 16-byte match
        ŌåÆ recovered (stateA, stateB)
```

#### Implementation

I built the A-side table in chunks of 2^22 states to keep memory reasonable, using 8-byte tags for compact storage with full 16-byte verification on tag collisions. The B-side probing ran in parallel across CPU workers.

One pitfall: the key packing format. I had to try two variants:

- `dup=False`: `LE64(state) || 8├Ś0x00` (the correct one)
- `dup=True`: `LE64(state) || LE64(state)` (incorrect but worth trying)

## Full Exploit Script

```python
from Crypto.Cipher import AES

keyA = bytes.fromhex("13cdb201000000000000000000000000")
keyB = bytes.fromhex("5cb47902000000000000000000000000")

def dec_block(c):
    y = AES.new(keyB, AES.MODE_ECB).decrypt(c)
    return AES.new(keyA, AES.MODE_ECB).decrypt(y)

def dec_all(ct_hex):
    ct = bytes.fromhex(ct_hex)
    assert len(ct) % 16 == 0
    return b"".join(dec_block(ct[i:i+16]) for i in range(0, len(ct), 16))
```

## Key Takeaways

- **Key derivation can silently destroy entropy.** The malware reads 32 bytes from `/dev/random` and runs it through HMAC-SHA3-256 ŌĆō all sounds secure. But one right-shift by 38 bits collapses 128-bit keys to 26-bit keys. Always trace the full key derivation path.
- **Double encryption doesn't double the security.** Double AES with independent keys should provide 2^256 security. MITM reduces it to 2├Ś2^128. But when each key only has 26 bits, MITM turns it into 2├Ś2^26 ŌĆō trivially brute-forceable.
- **ECB mode enables block-wise attacks.** Each 16-byte block encrypts independently, so you can attack one block at a time with known plaintext. CBC or GCM would have made this significantly harder.
- **Protocol framing gives you known plaintext.** The `de c0 de c0 ff ee` magic header + `REQCONN_OK` string are predictable. Real protocols should avoid fixed plaintext in encrypted streams, or at minimum use authenticated encryption modes.
