## The Storyline

The solved tasks chain together into one coherent intrusion:

1. **Task 1** — A rogue OpenRC service (`eanzovmwru`) on an Alpine dev box reveals a C2 callback config, but no actual malware on disk.
2. **Task 2** — A PCAP shows the missing piece: a compromised router (Router 3) poisoning DNS for `archive.ubuntu.com`, redirecting victims to a custom C2 over a π-themed port pair.
3. **Task 3** — A router memory dump exposes the malware running *fileless* via `memfd_create()`, with a custom base64 + rolling-XOR payload that hijacks 35 package/PyPI domains.
4. **Task 4** — The implant itself is a two-stage "meme packer" with heavy anti-debug; patching jumps (not calls) preserves its RC4 state and reveals the drop path `/tmp/.Q4JMdcyorlib52mg`.
5. **Task 5** — The downloaded module uses double AES-128-ECB, but a key-derivation bug collapses each key to 26 bits — a meet-in-the-middle attack recovers the C2 URL (a Mattermost instance).

A recurring tell throughout: the adversary consistently uses RFC 5737 documentation ranges (`203.0.113.0/24`, `198.51.100.0/24`) for its infrastructure.

## Tools Used

- **mount / file / sha1sum** — read-only ext2 forensic mounting and hashing (Task 1)
- **Wireshark / tshark** — PCAP protocol analysis and DNS/FTP inspection (Task 2)
- **Volatility 3 / dwarf2json** — Linux memory forensics with a custom kernel symbol table (Task 3)
- **Ghidra / IDA** — disassembly and decompilation of the malware stages (Tasks 3–5)
- **GDB** — dynamic analysis, jump patching, and in-memory payload dumping (Task 4)
- **OpenSSL libcrypto / PyCryptodome** — AES-128-ECB primitives and decryption (Task 5)
- **Python** — custom base64/XOR decoders and the meet-in-the-middle key search (Tasks 3, 5)
- **gzip / xz / brotli** — decompressing the provided dump, kernel, and symbol map (Task 3)
